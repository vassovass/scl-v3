Webhook Configuration (for PRD 76)

Webhooks notify StepLeague about payment events. Configure the URL now so it's ready when PRD 76 implements the handler.

Step 6.1 — Set Webhook URL

  1. Dashboard → Settings → API Keys & Webhooks
  2. Scroll to Webhook URL
  3. Enter: https://[your-domain]/api/webhooks/paystack
For local dev: use ngrok (npx ngrok http 3000) and set the ngrok URL. For Vercel preview: https://[branch].your-project.vercel.app/api/webhooks/paystack

Step 6.2 — Copy Webhook Secret

Below the webhook URL, find the Webhook Secret Key. Copy it to .env.local as PAYSTACK_WEBHOOK_SECRET. Used for HMAC signature verification via the x-paystack-signature header.

Gotchas & Warnings

!Name Matching — KYC Blocker

Paystack account name, ID document name, and FNB account name must all match exactly. Middle name present/absent, abbreviations, or nicknames cause rejection.

2SA Phone for OTP

Paystack sends OTP to your SA number during signup and dashboard actions while you're in Vietnam. Options: best — SA SIM with international roaming · alternative — VoIP receiving SA SMS · fallback — email support@paystack.com before starting.

3Vietnamese IP is Fine

Paystack verifies your business based on registration country (South Africa), not physical location. No VPN needed. Logging in from Vietnam works.

4ZAR Settlement Only

Paystack always settles in ZAR to your FNB account. The PRD 74 schema's amount_cents field stores ZAR cents for Paystack transactions. Tier prices shown to users in USD are a display concern — PRD 76 must decide how to handle the display price (USD) vs settlement amount (ZAR) split.

!Two-Step Subscription Flow — Architecturally Significant

Paystack requires at least one completed transaction before a subscription can be created (subscriptions use card authorizations from prior transactions). This differs from Stripe/Paddle where first payment + subscription creation are atomic.

PRD 76 checkout flow must be:

  1. Initialize a one-time transaction (first payment)
  2. Customer completes payment on Paystack checkout
  3. charge.success webhook fires with authorization_code
  4. Backend creates subscription using that authorization

6Sole Proprietorship Collection Limits

Higher limits than Starter, but still capped. Monitor in the dashboard. When approaching the limit: Compliance → Profile → change to Registered Business → upload CIPC certificate and director info.

7Test Mode ≠ Live Mode

Completely separate environments. Plans, customers, subscriptions, and keys in test mode don't exist in live mode. When switching to production: re-create all subscription plans, update .env to pk_live_ and sk_live_ keys, update webhook URL.

8No Paystack MCP Server

Unlike Paddle and Stripe, Paystack has no MCP server. All billing management is via Dashboard UI or direct API calls. A lightweight MCP could be built later when needed.

9Webhook Idempotency

Paystack retries webhooks on non-200 responses, so the same event may arrive multiple times. The PRD 76 handler must be idempotent — use external_subscription_id and external_payment_id as deduplication keys.

10Bank Confirmation Letter Expiry

The FNB confirmation letter must be less than 6 months old at KYC submission time. If delayed, generate a fresh one via the FNB App.

Provider Adapter Pattern

All Paystack-specific code lives in a provider adapter to preserve the migration path to Paddle:

src/lib/payments/paystack.ts   ← Paystack-specific API calls
src/lib/payments/types.ts      ← Provider-agnostic interfaces

The league_subscriptions.metadata JSONB column stores Paystack-specific data (plan codes, authorization codes), keeping core columns provider-agnostic.
